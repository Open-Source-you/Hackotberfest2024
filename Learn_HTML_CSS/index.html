Headings

Headings should be used in order, starting with h1. If anything looks like a heading, it should be marked up as a heading. Note: Correct use of headings are important for screen-reader use. Screen-readers come with shortcut keys to navigate via headings and is commonly used by sighted users.

Complex data tables

Complex tables should use the scope attribute to associate data cells with column headers. Scope can also be used to associate row headers with data cells.

Layout table

Where possible, tables should not be used for layout. However if a table is used for layout because of a widget library or the support of legacy code, the table element should have a role attribute of presentation, and should NOT have a and tags or scope and header attributes.

Labels

Every form input should have a label. The label tag should have a for attribute. This applies to input (all types), select and textareas. Where a label is not possible use the placeholder and title attribute.

Required fields

A mandatory field should have the required attribute. The label for a mandatory input should have either the word required or an asterisk to indicate that the field is mandatory. If an asterisk is used, add some wording to explain what the asterisk represents should be presented to the top of the form. E.g. * indicates required field

Textarea

If a textarea has maximum character limit (which it should) this must be specified in the label. The textarea should also set the value to the maxlength attribute.

Date Fields

Date fields should show the format in the label. If you are using HTML5 consider using the input type="date".

Tooltip

An accessible tooltip should:

  • appear on focus, not just mouseover
  • have the aria-describedby attribute on the element which points to the tooltip container

Should be in the format dd/mm/yyyy
